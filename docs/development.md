# Development Guide

This guide covers the local development workflow for Safe Code.

## Requirements

- Node.js compatible with the TypeScript and VS Code extension dependencies.
- VS Code for running and debugging the extension.
- npm for installing dependencies and running scripts.

## Install

```bash
npm install
```

## Compile

```bash
npm run compile
```

This runs `tsc -p ./` and writes compiled JavaScript to `out/`.

## Watch Mode

```bash
npm run watch
```

This keeps TypeScript running in watch mode while you edit files.

## Running In VS Code

Open this repository in VS Code and start an extension development host from the Run and Debug panel. The extension entry point is `out/extension.js`, which is generated by `npm run compile`.

If you add a launch configuration later, keep it focused on compiling TypeScript and launching the extension development host.

## Manual Test Cases

Create or open a supported file inside the workspace and try these examples.

This should warn:

```ts
const apiKey = "sk_live_123456789abcdef";
```

This should warn:

```dotenv
DATABASE_URL=postgres://user:pass@example.com/app
```

This should warn:

```text
-----BEGIN PRIVATE KEY-----
```

This should not warn because it is a placeholder:

```ts
const exampleApiKey = "your-api-key-here";
```

## Common Changes

To add a detection rule, edit `src/rules.ts` and follow [Detection Rules](./detection-rules.md).

To change which files are scanned, edit `supportedExtensions` or `.env` handling in `src/scanner.ts`.

To change default skipped paths, update `defaultIgnoredPaths` in `src/scanner.ts` and the `safeCode.ignoredPaths` default in `package.json`.

To change diagnostics behavior, edit `scanNow(document)` in `src/extension.ts`.

To change quick fixes, edit `SafeCodeActionProvider` in `src/extension.ts`.

To change ignore behavior, edit `src/ignoreStore.ts`.

## Future Features

The next natural feature is a full workspace scan command. It should use VS Code workspace APIs to find supported files inside the current workspace, respect ignored paths, open each document, scan it, and write diagnostics to the same diagnostic collection.

Another useful feature is severity mapping. Rules already have internal `low`, `medium`, and `high` severity values, but the MVP maps all findings to `DiagnosticSeverity.Warning`.

Project-level ignore configuration can be added later if teams need shared ignores. The current ignore storage is intentionally local so the MVP does not write project files.

The smart `.env` migration feature should be added after the scanner and ignore UX are stable.
